# Gap Audit — 2026-07-01

Five parallel deep-dives (ML/DS rigor, data authenticity, business narrative,
frontend demo-ability, engineering credibility) against the target audience:
**Consulting/Ops + ML/DS hiring managers at top firms.** Every finding is grounded
in a specific file. Ranked by how much it stops the project from being genuinely
impressive.

---

## The one-sentence problem

**This is an impressive software-engineering portfolio wrapped around ML/OR that
mostly doesn't survive scrutiny** — the models predict deterministic functions of
their own inputs, every headline metric is measured on data or a config different
from what's actually served, and a confident "everything is real, nothing is
fabricated" framing sits directly on top of synthetic targets. The P0–P5 pass fixed
the *packaging* (deploy, MLflow plumbing, dollar-framing docs); it did not fix the
*science*.

The tell that hurts most: the "demand is REAL, not synthetic" banners sitting on top
of synthetic data read as either not understanding leakage or knowingly overselling —
worse than saying nothing. **Every quantitative claim on the resume is currently a
question the candidate can be caught on.**

## TIER 0 — Credibility blockers (these end interviews)

**0.1 — Every "ML model" predicts a deterministic function of its own inputs (no learning / leakage).**
- Lead-time model: target = `base_days(cat)·(1+1.5·macro_stress)·(1+dist_km/20000)`, no
  noise, `macro_stress` constant across rows; both drivers are features →
  Ridge/RF/GBT/MLP "bake-off" fits a closed-form equation, R²≈1.0 by construction.
  `backend/app/ml/lead_time_model.py:153-165`, `seeds/train_ml_models.py:45,111`.
- Regime classifier: label = `(capacity_util≥75)&(inventory_ratio≤1.35)` while the same
  series are fed as features → recall≈1.0 tautology. `backend/app/ml/fred_client.py:87-111`.
- No observed target exists in the DB for either; the labels were invented and then "predicted."

**0.2 — The deployed DB serves synthetic random-noise demand while claiming "real FRED demand."**
- Reconstructed shapes vs real FRED: corr = **−0.11** (would be ~1.0 if real); DB std 0.13
  vs FRED 0.03 (4× noisier = random). The shipped DB (`backend/supply_chain.db`, Jun 16)
  predates the "real FRED" rewrite (`fbc1930`, Jun 18) and was **never regenerated**.
  Generator: `train_forecasts.py` @ git `778eb6c` `rng.normal(...)`. Model docstring even
  admits "52 weekly **synthetic** drawdown rows." Directly violates the real-data-only rule.

**0.3 — Even the "fixed" demand generator isn't real per-part demand.**
- `demand = (total_stock/52)·risk_multiplier·index_shape`, no noise; `index_shape` is ONE
  macro IPG3344S curve reused for all 791 parts. So all 791 "series" are the same curve ×
  a scalar, and demand is derived from current inventory (economically backwards).
  `seeds/train_forecasts.py:96-125,169`.

**0.4 — Headline forecast metric is a bait-and-switch.**
- Advertised "Prophet WAPE 4.8% vs seasonal-naive 8.7%, +45% skill" is measured on
  **monthly macro FRED, yearly-seasonality ON**. The **served** forecaster is **weekly,
  per-component, seasonality OFF (trend-only)**. Deployed forecasts are never backtested.
  n=1 series, 3 origins, 36 points, no CIs — "+45%" is one split.
  `run_forecast_backtest.py:72-77` vs `train_forecasts.py:194-202`.

**0.5 — The flagship "graph-aware optimizer" benchmark is a structural null: +0.00% on every BOM.**
- `cascade_risk_score` is computed from component IDs only, never the chosen distributors →
  bit-identical across graph-aware vs baseline arms. Cost surcharge capped at 15% so it
  never overcomes real price gaps → cost identical too. The only non-zero column (ETA ±0.1d)
  is **unseeded `random.gauss`** noise, while the artifact header falsely claims "Seed: 42."
  `graph/simulation.py:69-76`, `sourcing.py:406-418`, `solve.py:80-92`,
  `backend/BENCHMARK-RESULTS.md`. The central differentiator produces identical output to
  the version without it.

**0.6 — There is no optimizer-vs-naive-baseline benchmark — the one comparison an interviewer demands.**
- Both benchmark arms run the *same* MILP with only `graph_aware` toggled. "Could I get the
  same answer by sorting offers by price and picking cheapest per part?" is unanswered
  anywhere. `run_benchmark.py:347`. (The MILP itself is real and non-trivial — genuine
  fixed-charge sourcing — its value over greedy is just never shown.)

**0.7 — The interview-guide "wow" numbers don't reproduce from the real system.**
- Guide cites Fiedler 0.05 (real run falls back to **0.0**, ARPACK non-convergence), DigiKey
  "40% of offers" (real: **11.2%**), "+15% cost / 12 orphaned" (real: ~0% delta, **1**
  orphaned); "15%" is just the hardcoded `EMERGENCY_COST_PREMIUM` constant surfacing.
  `docs/RESILIENCE_INTERVIEW_GUIDE.md`. The candidate is coached to say numbers the live demo
  contradicts.

**0.8 — Hardcoded fake per-supplier numbers in the flagship Resilience impact table.**
- Every "alternative supplier" row is built as `{lead_time_days: 21, cost_delta_pct: 5}` —
  literal constants, identical for all three scenarios. Expand two rows in a demo → obviously
  fabricated. `frontend/src/pages/ResiliencePage.tsx:352-356,437-441,522-526`.

## TIER 1 — Major substance/credibility gaps

**1.1 — Risk-measure illiteracy red flags (+ a real bug).**
- "EVaR-95" is actually CVaR/Expected-Shortfall; "SIR-style cascade" is one-shot percolation
  (no propagation/time/recovery); per-node failure probability = min-max-normalized
  betweenness centrality (category error — forces one supplier to p=1.0, one to p=0.0 as an
  artifact); failures are independent draws (no correlation → anti-conservative tail).
  **Bug:** `fulfillment_rates.sort()` mutates in place before being zipped against unsorted
  `cost_inflations` → corrupts the tail pairing of the headline metric.
  `graph/simulation.py:38,130-136,146-150,171,182-186`.

**1.2 — Academic citations describe programs not actually solved.**
- "Weighted-sum multi-objective scalarization" (Marler & Arora 2004) cited, but
  `w_cost/w_time/w_carbon` never appear in `sourcing.py` — the MILP minimizes cost only;
  strategies differ by 3 ad-hoc knobs. Cross-dock labeled "Lagrangian relaxation of the CFLP"
  (Daskin 2013) but is a brute-force min over 10 hubs, no multipliers, no capacity.
  `strategies.py:4-5`, `cross_dock.py:12-14,176-183`.

**1.3 — Chronos benchmark almost certainly wasn't really run.**
- 0.25s load + 0.01s inference for a torch TSFM on CPU is physically implausible. The
  cold-start "Chronos beats Prophet" win is manufactured by running Prophet with
  `yearly_seasonality=True` on 6 months of data (guaranteed to blow up); seasonal-naive also
  beats that config. Chronos only *ties* naive. `docs/chronos_benchmark.json`,
  `run_chronos_benchmark.py:71,381`.

**1.4 — No sharply-defined customer, decision, or dollar story.**
- Only framing is a hobby-scale 5-component BOM for 50 IoT units; PROJECT.md states the
  customer is literally the recruiter. Dollar-framing is proxy-on-proxy (WAPE from a national
  *production index*, not part demand) with trivial magnitude (~$3.7k/yr per $1M spend); the
  optimizer-savings tile is ~$0 (see 0.5). No before/after $ number survives a partner poking
  the assumptions. `interview-walkthrough.md`, `IMPACT_FRAMING.md`.

**1.5 — Outputs numbers/delta-cards, not recommendations.**
- No recommendation engine, no ranked dual-sourcing plan, no sensitivity/tornado analysis.
  Consulting/ops buy decisions, not dashboards. "Here's the cost of de-risking" is half an
  answer; "de-risk Arrow first, 8× ROI vs DigiKey" is the deliverable.

**1.6 — "Nexar/Octopart API" data is really an anonymous HuggingFace re-upload.**
- `seed_db.py:150` loads `mdnh/electronic-components-supply-chain` (anonymous uploader, no
  date/license/citation). `seed_live.py` has a real NexarClient path but needs keys and wasn't
  used. README markets "Nexar/Octopart API (real component pricing)." Provenance/freshness
  unverifiable. Also: MOQ uniformly = 1 (no price breaks); README says 8,731 offers, DB has
  **8,176**; ~40 Asian distributors share one hardcoded Shenzhen coordinate.

**1.7 — Two of four "live risk feeds" are dormant in the deployment.**
- GPR + PortWatch fetch live; **ACLED + FRED** need keys absent from `.env`/`render.yaml` →
  silently return None. README markets a "four-layer risk-sensing architecture."
  `app/feeds/fetchers.py`. (To its credit, feeds degrade honestly, no fake fallbacks.)

**1.8 — MLflow champion registry is decorative w.r.t. serving.**
- `select_champion` genuinely searches/registers/aliases, but inference loads in-process
  joblib and never resolves the `champion` alias. And it selects between 4 models scoring
  ~perfect on leaked targets → discriminating on noise. `mlflow_tracking.py:216-271`,
  `costs.py:118-133`.

## Fix plan (sequenced by impact-per-effort)

### The pivotal decision (forks everything below)
The deepest gap — models don't learn because there are no real observed targets — has two
routes:
- **Route A "Make it real":** hunt for genuine observed targets (real per-SKU/category demand
  or shipment history; real observed lead times; real shortage/allocation events) and retrain.
  Highest payoff, but depends on such datasets being findable.
- **Route B "Make it honest + repoint the headline":** relabel the rules-based estimators
  honestly (drop the leaked-target R² claims), scope the forecast metric correctly, and move
  the quantitative headline onto the genuinely-strong OR (MILP-vs-greedy) + graph work. Faster,
  fully in our control, and removes every fabrication trap.

Most likely best: **Route B as the spine, Route A opportunistically** where real data exists
(demand is the best candidate).

> **DECISION (2026-07-01): Route B chosen** — honest relabel + repoint the quantitative
> headline onto the OR (MILP-vs-greedy) + graph work, with Route A layered in
> opportunistically (real per-part demand is the top Route-A target).

### Phase 1 — Defuse the fabrication traps (highest priority, ~1-2 days)
1. Regenerate/replace the served DB so data matches claims (or fix the claims). Kill the
   "demand is REAL / nothing fabricated" banners; replace with honest scoping language.
2. Re-derive every interview-guide number from an actual run; pin the demo to one BOM whose
   real output is defensible. Fix the Fiedler 0.0 fallback.
3. Remove/relabel the hardcoded 21d/+5% alternatives in `BOMImpactTable`.
4. Fix the `fulfillment_rates.sort()` tail-pairing bug.

### Phase 2 — Build the ONE benchmark that matters + fix risk literacy (~2-4 days)
5. Optimizer vs naive cheapest-offer greedy baseline → a real, defensible cost/service/risk
   delta. This is THE consulting artifact.
6. Make graph-awareness actually change selection (disruption-conditioned scenarios) or reframe
   it honestly as "insurance that costs ~0% in normal times, changes sourcing under
   disruption" and prove *that*. Make `cascade_risk` depend on chosen distributors.
7. Rename EVaR→CVaR (or implement real EVaR); SIR→percolation (or implement real cascade);
   replace centrality-as-failure-prob with a real hazard model; add correlated regional shocks.

### Phase 3 — Make at least one model genuinely learn (Route-A dependent, ~variable)
8. Best candidate: real per-SKU/category demand or shipment series → honest per-part backtest
   of the *served* forecaster. Otherwise relabel ML models as rules-based estimators and let
   the correctly-scoped Prophet-vs-naive backtest + the OR/graph work carry the quantitative
   story.

### Phase 4 — Business framing (~1-2 days)
9. Anchor to one sized, named scenario (e.g. a contract manufacturer sourcing a $50M/yr
   multi-BOM electronics portfolio, quarterly re-sourcing, China/Taiwan concentration). Tie
   every metric to that P&L; express EVaR/CVaR in absolute dollars.
10. Add a recommendation layer (rank critical nodes by risk-reduction-per-dollar → prioritized
    dual-sourcing plan) + a one-way sensitivity/tornado chart.
11. Add a concrete tariff/reshoring scenario tied to real component origins.

### Phase 5 — Engineering & demo polish (~1 day)
12. DB/models out of git history (build at deploy / LFS). Add ruff + mypy + coverage to CI.
    Move graph/resilience logic into `services/`. Provision ACLED/FRED keys on Render (or
    downgrade the "4 feeds" claim). Sync test counts, remove `print()` in auth. Re-run the
    benchmark to a clean-delta screenshot; backport Dashboard polish to the plain pages.

## Bottom line
A senior DS interviewer leans forward briefly (real MILP, real tests, thoughtful Chronos
reasoning) then loses trust fast (leaked targets, un-run benchmarks, mislabeled risk measures).
A McKinsey/Amazon-Ops interviewer loses interest at "so what?" (hobby-scale hypothetical, 0.00%
flagship benchmark, demo numbers that don't reproduce). The path to genuinely impressive is
**not more features** — it's real observed targets where possible, an optimizer-vs-greedy dollar
benchmark, honest risk-measure names, headline metrics that measure the model actually served,
and one sized business scenario that turns the dashboard into a ranked recommendation.
