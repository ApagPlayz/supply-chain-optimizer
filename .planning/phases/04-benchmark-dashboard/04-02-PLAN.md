<objective>
Build the three backend benchmark endpoints — `/benchmark/summary`, `/benchmark/fiedler-curve`, and `/benchmark/cascade-heatmap` — that the frontend (04-03) and map page (04-04) consume. The summary endpoint aggregates paired A/B rows from `optimization_runs` for the latest `run_id`, computing percentage deltas across cost, ETA, CO2, and cascade risk. The Fiedler curve endpoint reads the pre-computed `GraphState.fiedler_curve` from the Phase 2+Phase 4 lifespan block. The cascade heatmap endpoint produces per-distributor BOM-collapse probability arrays for maplibre rendering.

Purpose: Give the frontend a single authoritative API surface for all benchmark data; the dashboard cannot render until this plan is complete.
Output: `backend/app/api/benchmark.py` (router + 3 endpoints + Pydantic schemas), registration in `__init__.py`, 12+ tests in `test_benchmark_api.py`.

**Endpoint implementations:**

`GET /benchmark/summary` (optional `?run_id=N`):
1. Query `SELECT DISTINCT run_id FROM optimization_runs ORDER BY run_id DESC LIMIT 1` to find latest.
2. If `run_id` param provided, use it; validate rows exist for that run_id or raise 404.
3. Load all rows for that run_id from `OptimizationRun` (expected: 20 rows).
4. Partition into `baseline_rows` (graph_aware=False) and `graph_aware_rows` (graph_aware=True).
5. If either partition is empty, raise 404 with detail: `"No benchmark rows found for run_id={X}. Run: python -m seeds.run_benchmark"`.
6. Compute per-BOM deltas, then aggregate means. Delta sign convention: negative = graph-aware cheaper/faster/less risky (a win).
7. For Monte Carlo: aggregate `eta_p10_days`, `eta_p50_days`, `eta_p90_days`, `mc_evar_95` across BOMs per group.
8. Tradeoff: find the BOM where graph_aware is WORST on any single axis (highest positive delta). If all deltas negative, pick closest-to-neutral (smallest absolute negative delta). Set `losing_axis` and `narrative` string using the UI-SPEC copywriting template: `"{bom_name}: graph-aware is +{X}% {axis} because the cheapest distributor carries a k-core component and graph-aware routes around it."` — replace "k-core" with "high-centrality" when actual k-core data is unavailable from `selected_distributor_names`.
9. `feeds_fallback`: True if any row's `feeds_available` JSON contains a False value.
10. `noise_floor_pct`: hardcoded 2.0 (UI uses `abs(cost_delta_pct) < noise_floor_pct` to trigger low-confidence banner).

`GET /benchmark/fiedler-curve`:
1. Call `_require_graph_state()` (same helper pattern as graph.py, local import to avoid circular).
2. Read `gs.fiedler_curve` (list of dicts from Phase 2+Phase 4 pre-compute at lifespan).
3. If empty list, raise HTTPException(status_code=503, detail="Fiedler curve not computed — check server startup logs").
4. Baseline point is `gs.fiedler_curve[0]` (step=0, removed=None, removed_name=None).
5. Map each entry to `FiedlerPoint`. The `collapsed_boms` field: read from `entry.get("collapsed_boms", [])` — `run_benchmark.py` writes this field during pipeline execution; default to `[]` if absent (graph state only, no pipeline run yet).
6. Return `FiedlerCurveResponse(points=[...], baseline_lambda2=gs.fiedler_curve[0]["lambda2"])`.

`GET /benchmark/cascade-heatmap`:
1. Call `_require_graph_state()`.
2. Query `SELECT DISTINCT run_id FROM optimization_runs ORDER BY run_id DESC LIMIT 1`. If no rows, return empty list (NOT 404 — cascade is optional overlay).
3. Load all rows for latest run_id. For each unique distributor that appears in any row's `selected_distributor_ids`, compute: `weight = count_of_runs_where_this_distributor_failed_and_bom_collapsed / total_runs`. Since we do not have per-distributor failure data in the current schema, use `cascade_risk_score` as a normalized weight proxy: `weight = mean(cascade_risk_score)` across all rows where that distributor was selected (from `selected_distributor_ids`). Normalize to [0, 1] by dividing by max weight across all points.
4. Join distributor IDs to `Distributor.latitude/longitude/name` via DB query.
5. Return `CascadeHeatmapResponse(points=[HeatmapPoint(...) for each distributor with weight > 0])`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → API | `?run_id=N` query param is user-controlled integer |
| API → DB | SQLAlchemy query on optimization_runs (read-only) |
| API → GraphState | In-process read of pre-computed singleton (no I/O) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-02-01 | Tampering | `?run_id=N` query param | mitigate | FastAPI auto-validates `run_id` as `Optional[int]` via Pydantic; non-integer values return 422 automatically; range unbounded but 404 returns on missing run_id — no data leakage |
| T-04-02-02 | Denial of Service | `/benchmark/summary` response size | mitigate | Response capped at 20 rows per run_id (10 BOMs × 2 graph_aware is the benchmark schema contract); no unbounded query; `bom_deltas` list max length = n_boms (bounded by run_benchmark.py output) |
| T-04-02-03 | Information Disclosure | Benchmark exposes distributor names + cost deltas | accept | All data is aggregate analytics with no user PII, no individual prices, no authentication tokens. Mirror of `/graph/metrics` disposition (T-02-04 in Phase 2): public aggregate data with no sensitive fields |
| T-04-02-04 | Elevation of Privilege | Unauthenticated endpoint accessed by unauthorized users | accept | Per CONTEXT.md D-05 pattern and graph.py precedent: benchmark tab is public dashboard data. No user credentials, no mutation paths, no admin functions. ASVS V4 assessment: accept disposition (public read-only analytics) |
| T-04-02-05 | Spoofing | Server startup with no GraphState (graph build failed) | mitigate | `_require_graph_state()` raises HTTP 503 with clear diagnostic message when `get_graph_state()` returns None — prevents silently returning empty/stale data |
</threat_model>
